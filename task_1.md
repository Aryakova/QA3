# Многозвенная клиент-серверная архитектура

## Основные компоненты и их назначение

| Компонент                | Назначение |
|--------------------------|------------|
| Клиентское приложение (Client) | Взаимодействие с пользователем, отправка запросов к серверу, отображение данных. |
| Веб-сервер (Web Server)         | Приём HTTP(S)-запросов от клиентов, передача их на обработку в бизнес-логику, возврат ответов клиенту. |
| Сервер приложений (Application Server) | Обработка бизнес-логики приложения, выполнение вычислений, обработка данных, взаимодействие с другими сервисами. |
| Сервер баз данных (Database Server)    | Хранение, обработка и предоставление данных по запросу серверов приложений. |
| Балансировщик нагрузки (Load Balancer) | Распределение входящих запросов между несколькими серверами для повышения производительности и отказоустойчивости. |
| Кэш-сервер (Cache Server)              | Временное хранение часто используемых данных для ускорения доступа и снижения нагрузки на БД. |
| Службы очередей (Queue Services)       | Асинхронная обработка задач, обмен сообщениями между компонентами системы. |

# Отказоустойчивость: «Горячий» и «Холодный» резерв

| Характеристика         | Горячий резерв                | Холодный резерв                |
|------------------------|-------------------------------|-------------------------------|
| Определение            | Резервная система постоянно работает в синхронном режиме с основной и готова к немедленному переключению. | Резервная система выключена или неактивна, включается только при отказе основной. |
| Готовность             | Максимальная, резерв всегда готов к работе. | Минимальная, требуется время на запуск и инициализацию. |
| Синхронизация данных   | Постоянная, данные синхронизируются в реальном времени или с минимальной задержкой. | Может отсутствовать или выполняется периодически, возможна потеря данных. |
| Время переключения     | Практически мгновенное (секунды). | Заметная задержка (минуты и более). |
| Стоимость              | Высокая (дублирование ресурсов, энергозатраты). | Низкая (ресурсы используются только при сбое). |
| Плюсы                  | Минимальное время простоя, высокая доступность. | Экономия ресурсов, простота реализации. |
| Минусы                 | Высокие затраты, сложность поддержки. | Долгое восстановление, возможна потеря данных, ниже доступность. |
